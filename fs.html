The Origin Story

Why filesystems exist — from punch cards to NVMe, the full history with context

Imagine it's 1950. You're a programmer at IBM. Your "computer" fills an entire room. There is no keyboard, no monitor, no disk. You write your program on paper tape or punch cards. You hand the stack to an operator. They feed it in. Hours later, you get a printout back.

There are no "files". There is no "storage". Every computation is ephemeral. When power goes off, everything vanishes. This is the world before filesystems.

1950s — MAGNETIC DRUMS & TAPE
The First "Persistent Storage"
Magnetic drums spin at ~3,000 RPM. Data encoded as magnetic spots. Programs stored on magnetic tape reels. No random access — you had to wind the tape to the right spot. To "find a file" you literally fast-forwarded tape like a cassette. Reading data out of order was impossibly slow.
💡 Insight: Sequential-only access forced early programmers to think in batches. This is why early filesystems were designed for sequential data — it matched the hardware.
1956 — IBM 350 RAMAC
The First Hard Drive — 5MB for $10,000/month
The IBM 350 RAMAC was the size of two refrigerators. It held 5 megabytes on 50 rotating platters. For the first time ever, you could read data in any order without rewinding tape. Random access was born. This one invention made modern filesystems possible.
💡 The key breakthrough: Random access means you can store an index of where things live, then jump directly there. Without random access, an index is useless. This is the fundamental idea behind every filesystem ever made.
1961 — Compatible Time-Sharing System (CTSS)
First Multi-User File Storage
MIT's CTSS at Bell Labs allowed multiple users to share one machine. Suddenly you needed to separate whose data was whose. CTSS introduced the concept of a user directory — a flat list of files per user. No nesting, no hierarchy — just a flat list. Simple but revolutionary.
1965 — MULTICS
Hierarchical Directories Invented
Bell Labs + MIT + GE built MULTICS (Multiplexed Information and Computing Service). MULTICS introduced the hierarchical directory tree — directories inside directories. Files described by a path from root. Segments (files) could be shared between users. Also introduced the concept of protection rings (kernel vs userspace). Most ideas in modern OS came from MULTICS. MULTICS was too complex and over-engineered, but it was the source code that Thompson and Ritchie mined when building Unix.
💡 Key innovation: The path string /usr/local/bin/python3 — that hierarchical naming — was born in MULTICS. Before this, you just had flat lists.
1969 — UNIX (Bell Labs)
Ken Thompson Writes Unix FS in 3 Weeks
After Bell Labs pulled out of MULTICS (too expensive, too slow), Ken Thompson and Dennis Ritchie built Unix. Thompson wrote the first Unix filesystem on a PDP-7 — a computer with 8KB of RAM. The design was beautifully simple: a superblock, inodes, data blocks. Everything is a file. Devices, pipes, sockets — all accessed with the same open/read/write/close API. This was a massive simplification from MULTICS.
💡 The genius of "everything is a file": By treating devices, pipes, and sockets as files, Thompson gave programmers ONE API to learn. cat /dev/sda reads a disk like a file. echo hello > /dev/tty writes to a terminal like a file. This philosophy runs through everything in Linux.
1974 — Unix V6, then V7
Inodes Refined, Long Filenames, Pipes
Unix Version 7 (1979) was the version that influenced everything — BSD, XENIX, System V, and eventually Linux. It had 14-character filename limits (due to the original inode structure), basic permissions, and the foundational tools: sh, awk, sed, grep, make. Bell Labs licensed it to universities. University of California Berkeley received it and started making their own improvements — BSD was born.
1983–1991
GNU Project + Minix = The Ingredients for Linux
Richard Stallman starts GNU (GNU's Not Unix) — free software tools. Andrew Tanenbaum writes Minix — a teaching OS with its own simple filesystem (MINIX FS, 64MB limit, 14-char filenames). A 21-year-old Finnish student named Linus Torvalds is frustrated by Minix's limitations. He starts writing his own OS kernel on an 80386 PC. He uses the MINIX filesystem at first because he needed something immediately.
1991 — Linux 0.01
Linux Born, Immediately Needs a Better Filesystem
Linus posts on comp.os.minix: "I'm doing a (free) operating system... just a hobby, won't be big and professional like gnu." The first Linux kernel supported MINIX FS — the only filesystem it could use. Within a year it was clear MINIX FS was a bottleneck. Rémy Card, a French kernel developer, designs the Extended Filesystem (ext). The rest is the history of each generation fixing the failures of the last.

How Storage Works Physically

What the filesystem is actually talking to — HDDs, SSDs, NVMe at the bit level

Before you can truly understand a filesystem, you must understand what's below it. The filesystem doesn't store bytes — it talks to a block device, which talks to a driver, which talks to hardware. The hardware has real physical constraints that shaped every design decision in every filesystem.

🔴 HDD (Spinning Disk)
⚡ SSD (NAND Flash)
🚀 NVMe

Hard Drive Anatomy

PlatterAluminum or glass disk coated with magnetic material. A drive has 1–8 platters stacked on a spindle. Data encoded as magnetic north/south pole = 1/0.
TrackConcentric circle on a platter surface. A 7200 RPM drive has thousands of tracks. Each platter surface = separate layer of tracks.
SectorThe smallest unit the drive can read/write. Traditionally 512 bytes (now 4096 bytes = "4K sectors" in Advanced Format drives). Cannot write half a sector.
CylinderSame track number across all platters. Cylinders were key for old CHS (Cylinder-Head-Sector) addressing — now replaced by LBA.
Read/Write HeadFlies nanometers above the platter surface on an air cushion. One head per platter surface. All heads move together on the arm.
Seek TimeTime to move arm to the right track. ~3-9ms. This is why random I/O kills HDD performance — each random access costs a seek.
Rotational LatencyAfter the head is positioned, wait for the disk to rotate the right sector under it. Average = half rotation. At 7200 RPM = ~4ms.
🎯

Why this matters for filesystems: On an HDD, writing data in one contiguous block is 10–100× faster than scattered random writes. This is why ext4 uses delayed allocation and block groups — to keep related data close together on disk, minimizing seek time. Fragmentation kills HDD performance because it forces extra seeks.

bash — Measure your HDD seek time
# Random read test — see how slow random I/O is vs sequential fio --name=seq_read --rw=read --bs=1M --size=1G --filename=/tmp/test # Sequential: ~120 MB/s for HDD fio --name=rand_read --rw=randread --bs=4k --size=1G --filename=/tmp/test # Random 4K: ~0.5 MB/s for HDD — 240x slower! # SSD random 4K: ~400 MB/s — no seek penalty # Check drive type and rotation speed cat /sys/block/sda/queue/rotational 1 # 1 = HDD, 0 = SSD hdparm -I /dev/sda | grep -i rpm Nominal Media Rotation Rate: 7200

NAND Flash — How SSDs Actually Work

NAND CellA floating-gate transistor that traps electrons to represent data. SLC=1 bit/cell (fastest, most durable). MLC=2 bits. TLC=3 bits (most SSDs today). QLC=4 bits (cheapest, least durable).
PageSmallest unit you can READ or WRITE on NAND. Usually 4–16KB. Smaller than a block. You must write whole pages.
Block (NAND)Group of 64–512 pages (~256KB–4MB). Critical: You can only ERASE an entire block, never individual pages. This is the most important constraint in SSD design.
Write AmplificationWhen you update 4KB of data, the SSD must: read the entire 256KB block into RAM, modify the 4KB page, erase the block, write the entire block back. That's ~64× more writes than you asked for.
Wear LevelingNAND cells die after ~1,000–100,000 erase cycles (TLC: ~1,000). The SSD controller spreads writes across all blocks evenly. Hot data + cold data get swapped to balance wear.
FTL (Flash Translation Layer)The SSD's onboard controller that presents a traditional block interface to the OS while managing NAND internally. It does wear leveling, garbage collection, bad block management, and the logical→physical mapping. It IS a tiny filesystem inside your SSD.
Trim (DISCARD)When Linux deletes a file, it tells the SSD "these blocks are free now" via TRIM. The SSD can pre-erase them during idle time so future writes are fast. Without TRIM, SSDs slow down over time.
⚠️

SSD design principles for filesystems: Because SSDs have no seek penalty, random and sequential I/O are roughly equal. But the erase-before-write constraint makes overwrites expensive. This is why Copy-on-Write filesystems (btrfs, ZFS) are especially good on SSDs — they never overwrite, always write to new locations, which is what NAND prefers naturally.

bash — SSD health and TRIM
# Check SSD health via SMART smartctl -a /dev/nvme0 | grep -E "Wear|Written|Health" # Enable TRIM (usually enabled by default on modern distros) systemctl enable --now fstrim.timer # weekly TRIM via systemd fstrim -av # manual TRIM all mounted FS # Mount options for SSDs — noatime stops unnecessary writes # In /etc/fstab: UUID=xxx / ext4 defaults,noatime,discard 0 1 # discard = enable TRIM on every delete (can slow things down — better to use fstrim timer) # Check if TRIM is supported lsblk --discard /dev/sda

NVMe — NAND at the Speed of PCIe

What is NVMe?Non-Volatile Memory Express. A protocol (not a hardware type) designed specifically for NAND flash, replacing SATA/AHCI which was designed for spinning disks. NVMe speaks directly over PCIe lanes to the CPU — no controller bottleneck.
Why SATA Was a BottleneckSATA supports 1 command queue with 32 slots. NVMe supports 65,535 queues with 65,535 commands each. SATA max = 600 MB/s. NVMe Gen4 = 7,000 MB/s. NVMe Gen5 = 14,000 MB/s. SSDs were being strangled by SATA.
Device Names/dev/nvme0n1 = first NVMe controller, namespace 1. /dev/nvme0n1p1 = first partition. Multiple namespaces allow the SSD to present multiple independent storage spaces.
Queue DepthNVMe can handle thousands of simultaneous I/O requests. This is why NVMe random IOPS (500,000+) are so much higher than SATA SSD (100,000) — not just the NAND, but the parallel queuing.
bash — NVMe inspection
# List NVMe devices nvme list # NVMe SMART health nvme smart-log /dev/nvme0 # Check queue depth and scheduler cat /sys/block/nvme0n1/queue/nr_requests cat /sys/block/nvme0n1/queue/scheduler [none] mq-deadline kyber # 'none' is often best for NVMe — it's fast enough to not need scheduling

The Problems a Filesystem Must Solve

Before understanding solutions, understand the hard problems

A disk is just a giant array of numbered blocks. Block 0, Block 1, Block 2, ..., Block 1,000,000,000. That's it. No filenames. No folders. No permissions. Nothing. A filesystem's entire job is to impose structure and meaning on this flat, dumb array. Here are the hard problems it must solve:

01
Naming — How do you find something?
The disk speaks in block numbers. You speak in names like /home/alice/docs/report.pdf. Something must translate that path into "read blocks 50234, 50235, 50236". This is the namespace problem. The solution: a hierarchical directory tree, each node mapping names → locations.
02
Free Space — Which blocks are available?
With millions of blocks, you need a fast way to find free ones. Options: a linked list of free blocks (slow to traverse), a bitmap (one bit per block — 0=free, 1=used, stored compactly), or a B-tree of free extents (btrfs/ZFS approach). ext4 uses bitmaps. btrfs uses a tree.
03
Metadata — What IS this file?
For each file: Who owns it? What permissions? How big? When created/modified? Which blocks contain the data? All of this must be stored somewhere. Unix solution: the inode — a fixed-size structure storing all metadata except the filename.
04
Reliability — What if power dies mid-write?
You're writing a file. It needs 3 operations: update inode, write data, update bitmap. If power dies after op 1, the filesystem is inconsistent. Solution options: journaling (log intent before doing it) or copy-on-write (never modify in place, always write new, then atomically update pointer). ext4 uses journaling. btrfs/ZFS use CoW.
05
Performance — How do you make it fast?
Files grow. Data fragments. The inode's 12 direct block pointers are full — now what? You need indirect blocks (pointers to blocks of pointers). But walking 3 levels of indirect blocks for a large file is slow. ext4's solution: extents — store ranges instead of individual pointers. btrfs uses a B-tree.
06
Concurrency — Multiple writers at once?
Two processes writing the same file simultaneously. Two threads allocating free blocks. The filesystem must use locks and transactions to prevent corruption. ext4 has a journal transaction system. ZFS uses atomic 256-bit transactions. btrfs uses copy-on-write which makes this naturally safe.

The fastest way to understand something is to build it yourself. Let's design a minimal filesystem.

You have a 10MB disk. It's a flat array of 2,560 blocks (each 4096 bytes = 4KB). You need to design a filesystem on it. Let's think through each decision exactly as the Unix designers did in 1969.

You need one fixed place the OS always knows to look for the structure of your filesystem. Unix called this the superblock. It lives at a fixed offset (block 1, leaving block 0 for the boot loader). The superblock contains: total blocks, inode count, block size, magic number (identify FS type), mount count, and where the inode table starts.

Step 5
Create the root directory
Inode 1 is always the root directory "/". A directory is just a file with type=directory whose data blocks contain a list of (inode_number, filename) pairs. When you do ls /, the kernel reads the data blocks of inode 1 and prints the filenames. When you do ls /home/alice/file.txt, it walks: look up "home" in root dir → get inode 45 → look up "alice" in inode 45 → get inode 203 → look up "file.txt" in inode 203 → get inode 891 → read inode 891's data blocks. Every path lookup is this walk.
Step 6
Handle large files — the indirect block problem
12 direct block pointers × 4KB per block = 48KB max file. Tiny! For larger files, you need indirect blocks. The indirect pointer points to a block full of more block pointers. At 4KB block size with 4-byte pointers: one indirect block holds 1024 pointers → 1024 × 4KB = 4MB. Double indirect: 1024 × 1024 × 4KB = 4GB. Triple indirect: 4TB. This is the old way. ext4 replaced it with extents — instead of storing individual block numbers, store a range: "blocks 1000 through 5000 are contiguous".

VFS — The Virtual Filesystem Switch

How Linux talks to 30+ different filesystems with one API

In 1985, Sun Microsystems had a problem: they wanted one Unix system to support both their local disk filesystem and NFS (Network File System) transparently. A program shouldn't need to know whether it's reading a local file or a file on a server 1,000 miles away. They invented the Virtual Filesystem Switch (VFS). Linux adopted it. Today Linux supports 30+ filesystems because VFS gives each one a standard interface to implement.

The VFS Object Model — 4 Key Structures

superblock object

Represents a mounted filesystem. Stores: device, block size, filesystem type, pointer to the root dentry, and a table of superblock_operations function pointers — including how to read/write inodes for this specific filesystem type.

struct super_operations {
  struct inode *(*alloc_inode)(...);
  void (*read_inode)(...);
  void (*write_inode)(...);
  void (*put_super)(...);
  int (*statfs)(...);
};

inode object

In-memory representation of a file or directory. The VFS inode exists in RAM while a file is open. It has generic fields (size, uid, permissions) plus a pointer to inode_operations — filesystem-specific functions for lookup, create, mkdir, rename, etc.

struct inode_operations {
  struct dentry *(*lookup)(...);
  int (*create)(...);
  int (*mkdir)(...);
  int (*rename)(...);
  int (*link)(...);
};

dentry object

Directory entry — maps a filename string to an inode. A dentry is created each time the kernel looks up a path component. They're cached in the dentry cache (dcache) — this is why repeated path lookups are fast. Dentries form a tree matching the directory hierarchy.

struct dentry {
  struct inode *d_inode;
  struct dentry *d_parent;
  struct qstr d_name; /* filename */
  struct list_head d_subdirs;
};

file object

Created when a process opens a file. Lives in the process's file descriptor table. Stores the current file position (offset), flags (O_RDONLY etc.), and file_operations — the read(), write(), seek(), mmap() implementations.

struct file_operations {
  ssize_t (*read)(...);
  ssize_t (*write)(...);
  loff_t (*llseek)(...);
  int (*mmap)(...);
  int (*open)(...);
};

Boot & Mount Sequence

Exactly what happens between pressing power and seeing a prompt — every filesystem step

1
BIOS/UEFI — Hardware Init
CPU starts executing from a fixed address. BIOS initializes hardware (RAM, PCIe, storage). UEFI reads the GPT partition table from disk sector 1. Finds the EFI System Partition (ESP, type 0xEF00, usually FAT32). Loads the bootloader from /EFI/ubuntu/grubx64.efi.
No Linux kernel loaded yet. No Linux filesystem driver exists yet. UEFI has its own FAT32 driver built-in. This is why /boot/efi must be FAT32 — it's the only filesystem UEFI guarantees it can read.
2
GRUB — Bootloader
GRUB (Grand Unified Bootloader) loads. It has its own ext4/btrfs/xfs read-only drivers compiled in. It reads /boot/grub/grub.cfg from the boot partition. Shows the OS selection menu.
GRUB locates the kernel image (vmlinuz) and initial RAM disk (initrd.img). Loads them into RAM. Passes kernel parameters (root device UUID, console settings). Jumps to kernel entry point.
3
Kernel Decompression + Early Init
The kernel image is compressed (gzip/lzma). It self-decompresses into RAM. Sets up basic memory management (page tables). Initializes the CPU. Detects hardware via device tree or ACPI tables.
4
initramfs — The Temporary Root Filesystem
The kernel mounts the initramfs (initial RAM filesystem) as the first root. This is a tiny compressed cpio archive containing just enough to mount the real root. It has: busybox tools, udev for device detection, filesystem modules (ext4.ko, dm-crypt.ko), RAID/LVM tools.
Why initramfs? The real root might be on LVM, software RAID, encrypted LUKS, or a network device. The kernel can't know in advance which modules to load. initramfs loads them dynamically based on what hardware is found.
bash — Inspect initramfs
# Extract and examine initramfs contents mkdir /tmp/initramfs && cd /tmp/initramfs zcat /boot/initrd.img-$(uname -r) | cpio -idv # See what's in there ls /tmp/initramfs bin/ conf/ etc/ init lib/ scripts/ sbin/ usr/ # The 'init' script is what runs first cat /tmp/initramfs/init
5
Switch Root — Mounting the Real Filesystem
initramfs's init script detects the root device (from kernel parameter root=UUID=xxx). Loads the right filesystem module. Runs fsck if needed. Mounts the real root at /sysroot. Calls switch_root to make it the new /.
bash — What happens during switch_root
# Inside initramfs init script (simplified): modprobe ext4 # load ext4 driver mount -t ext4 /dev/sda1 /sysroot # mount real root mount --move /dev /sysroot/dev # move devtmpfs mount --move /proc /sysroot/proc # move proc mount --move /sys /sysroot/sys # move sysfs exec switch_root /sysroot /sbin/init # pivot and exec init
6
systemd / init — Mounting Everything Else
PID 1 (systemd or traditional init) starts. Reads /etc/fstab. Mounts all remaining filesystems: /home, /var, /tmp, swap. Starts udev for device management. Launches services. Eventually presents a login prompt.
bash — Watch systemd mount units
# systemd creates mount units from /etc/fstab automatically systemctl list-units --type=mount # See boot timing — which mounts were slow? systemd-analyze blame | grep mount # See the full boot sequence as a tree systemd-analyze critical-chain

ext4 Block Groups — The Locality Trick

Why ext4 divides the disk into groups and why this is brilliant

The original Unix filesystem had one superblock, one inode table, one data region. Problem: if your file's inode is at block 5 and its data is at block 2,000,000, every file access requires two seeks millions of blocks apart. On an HDD, this is catastrophically slow. The BSD Fast Filesystem (FFS) in 1984 invented cylinder groups to solve this. ext2 copied this as block groups.

The Block Group Structure

🎯

Core idea: Divide the disk into fixed-size groups (typically 128MB per group in ext4). Each group has its OWN superblock backup, OWN inode table, OWN bitmaps, OWN data blocks. The filesystem tries to store a file's inode and its data blocks in the same group. This keeps everything local — one seek gets you the inode AND the data.

The Group Descriptor Table (GDT)

The GDT lives right after the superblock in Group 0. It's an array — one entry per block group. Each entry tells you where that group's bitmaps and inode table start. When you create a file:

  1. Find the parent directory → what group is it in?
  2. Try to allocate the new inode in the same group
  3. Allocate data blocks in the same group (or nearby)
  4. Update ONLY that group's inode bitmap and block bitmap
  5. Write data. Done — minimal cross-group I/O.

flex_bg — ext4's Optimization Over ext3

ext4 introduced flexible block groups (flex_bg). Instead of each group having its own inode table, the inode tables of several groups are consolidated into the first group of the flex_bg. This makes the inode tables themselves contiguous — large sequential writes, fewer seeks. The default flex_bg size in ext4 is 16 groups = 16 × 128MB = 2GB flex-group regions.

Extents vs Block Maps — The Critical Difference

Why ext4 uses extents and why it was a game-changer for large files

ext2 and ext3 stored file locations as a list of individual block numbers. A 1GB file with 4KB blocks = 262,144 block pointers. Reading that list takes time, wastes space, and makes large files slow. ext4 introduced extents: instead of saying "block 1000, 1001, 1002, 1003...", say "start at block 1000, length 4000 blocks". One extent record covers gigabytes of contiguous data.

Old Way — Block Map (ext2/ext3)

Each 4-byte entry stores ONE block number. To describe a 1GB file:

direct[0] = 50000
direct[1] = 50001
direct[2] = 50002
...
indirect → [50012, 50013, ..., 51036] (1024 entries)
dbl_indirect → [[51037...52060], [52061...53084]...]
262,144 pointers for 1GB file!

New Way — Extents (ext4)

Each 12-byte extent describes a RANGE. A 1GB contiguous file:

extent[0]: start=50000 len=262144
ONE ENTRY for the entire 1GB!

A fragmented file:
extent[0]: start=50000 len=10000
extent[1]: start=80000 len=50000
extent[2]: start=200000 len=202144
3 entries instead of 262,144

The Extent Tree

Extents are stored in a B+ tree structure inside the inode. The inode has 60 bytes for block pointers — in extent mode, this is the root of the extent tree. For small files (up to 4 extents), the tree fits entirely in the inode — no extra disk I/O needed. For larger, more fragmented files, the tree grows deeper with internal nodes stored in extra blocks.

Journaling — How Crash Safety Actually Works

The complete mechanics of ext3/ext4 journaling from write() to commit

Consider writing a file in ext2 (no journal). Three things must happen: (1) write data blocks, (2) update inode (new size, new block pointers), (3) update block bitmap (mark new blocks used). If power fails between steps 1 and 2: the data is written but the inode doesn't point to it — orphaned blocks. If it fails between 2 and 3: the bitmap shows blocks as free but the inode references them — double-allocation corruption. Without a journal, the only fix is a full fsck scan — which on a 1TB drive takes hours.

The Journal — A Write-Ahead Log

ext3/ext4 allocate a special file (or a dedicated partition) as the journal. The journal is a circular buffer of transactions. Before making ANY change to the filesystem, the kernel logs that change to the journal first. Only after the journal entry is safely on disk does the actual filesystem modification happen. On crash: replay the journal to complete interrupted transactions. Fast recovery — seconds, not hours.

1
BEGIN TRANSACTION
Kernel acquires journal transaction handle. Multiple operations are batched into one transaction for efficiency. Transaction gets a unique sequence number.
↓
2
LOG INTENT (Journal Write)
All dirty metadata blocks (inodes, bitmaps, superblock) are written to the journal log area first. In data=journal mode, data blocks go to journal too. In data=ordered (default), data blocks go directly to their final location but are flushed BEFORE the journal commit.
↓
3
JOURNAL COMMIT BLOCK
After all journal blocks are written, kernel writes a commit block with a checksum. This is the atomic moment. Once the commit block hits disk, the transaction is considered complete — even if power fails immediately after, recovery knows to replay it.
↓
4
CHECKPOINT — Write to Final Location
Asynchronously, the kernel writes the journaled metadata to their actual filesystem locations. This is called checkpointing. The journal space can be reclaimed once all blocks in a transaction are checkpointed.
↓
5
ON CRASH — Recovery
Boot: kernel mounts fs and checks if journal has uncommitted transactions (flag in superblock). If yes: replay committed transactions (safe to redo), discard uncommitted ones. 5-second recovery vs hours of fsck.

Three Journal Modes — The Speed vs Safety Tradeoff

data=journal (safest)

ALL data AND metadata go through journal. Double write for everything. Slowest. Safest. If power fails, NO data loss at all — not even partially written data. Used in databases requiring absolute durability.

Performance: ██░░░░░

data=ordered (default)

Only metadata goes through journal. Data blocks written to final location FIRST, then metadata journaled. Guarantees no stale data after crash but you might lose the last few seconds of writes. Best balance.

Performance: █████░░

data=writeback (fastest)

Only metadata journaled. Data written independently — might be out of order. After crash, a file might show new size but old (garbage) data content. Fastest. Use for scratch disks or where performance > consistency.

Performance: ███████

How btrfs and ZFS achieve crash safety without a journal, and why it enables magic features

Journaling solves crash safety by logging changes before making them. But it means writing everything twice. CoW takes a different philosophical approach: never modify data in place. Always write new data to a free location. Only then atomically update the pointer. If power fails at any point, the old data is still intact — the pointer was never updated.

Why CoW Makes Snapshots Free

📸

Snapshot = just saving the current root pointer. Since CoW never modifies old data, a snapshot is literally just recording which root node was the "filesystem state at time T". B-Trees in Filesystems

Why btrfs, ZFS, and XFS all use B-trees and how they work

ext2/3/4 used simple bitmaps and linear inode tables. Great for small filesystems, but as disks grew to terabytes with billions of files, linear scans became too slow. Modern filesystems like btrfs, XFS, and ZFS use B-trees — the same data structure powering every relational database — to manage everything: inodes, extents, checksums, free space. Understanding B-trees unlocks understanding of all modern filesystems.

Why B-trees for Filesystems?

O(log n)
Search, insert, delete all in logarithmic time. 1 billion files → only ~30 node reads to find any file.
Block-aligned
Each B-tree node is exactly one disk block (4KB). Perfectly aligned for disk I/O — no wasted reads.
Self-balancing
Splits and merges automatically. Performance stays consistent regardless of insertion order.

btrfs — "One B-tree to Rule Them All"

btrfs has one master B-tree called the chunk tree which maps logical address space to physical locations. Inside it are multiple sub-trees:

Root Tree
├── Chunk Tree — maps logical addresses → physical device extents
├── Extent Tree — tracks all allocated extents + back-references
├── FS Tree (per subvolume) — inodes, directory entries, file extents
├── Checksum Tree — CRC32C/xxHash for every data block
└── Device Tree — maps device IDs to block devices

Page Cache & Buffer Cache

Why Linux uses every byte of free RAM — and why that's the right call

Open top or free -h and you'll see Linux is using 95% of your RAM even with few programs running. New users panic. Experienced engineers smile. Linux caches everything it reads from disk in RAM — the page cache. Used RAM is better than wasted RAM. The kernel gives it back instantly when a process needs memory.

📖 Read Path — with Page Cache

  1. Process calls read(fd, buf, 4096)
  2. Kernel checks page cache for this file's page
  3. Cache HIT: return data from RAM immediately. Microseconds.
  4. Cache MISS: allocate a page in cache, read from disk (milliseconds), copy to process, cache stays warm for next access
  5. Same file read by 100 processes: only ONE copy in page cache, all processes share it

✍️ Write Path — Dirty Pages

  1. Process calls write(fd, buf, 4096)
  2. Kernel writes to page cache immediately — returns to process fast
  3. Page marked dirty (modified, not yet on disk)
  4. Background kernel thread (pdflush/kworker) writes dirty pages to disk
  5. Dirty writeback triggered by: time (30s), dirty ratio exceeded, explicit fsync()
🚨

The fsync() trap: write() returning success does NOT mean data is on disk. It's in the page cache. If power dies, data is lost. Call fsync(fd) to force flush to disk. Databases always fsync. Most apps don't need to — but knowing this explains many "data loss after crash" bugs.

Write-back & Delayed Allocation

The two performance tricks that make ext4 dramatically faster than ext3

ext3 allocated disk blocks the moment you called write(). This meant allocating blocks before the kernel knew the final file size — leading to fragmentation as files were allocated in small pieces. ext4 introduced delayed allocation: don't allocate real disk blocks until the page cache is actually being written to disk. By then, the kernel knows the full size and can allocate perfectly contiguous blocks.

ext3: Eager Allocation

  1. write() called with 4KB data
  2. IMMEDIATELY find free block, reserve it
  3. Repeat 1000 times → 1000 separate allocations
  4. Blocks scattered across disk → fragmentation
  5. Later writes are slow because of fragmented layout

ext4: Delayed Allocation

  1. write() called — data goes to page cache only
  2. Repeat 1000 times — all in page cache
  3. Writeback triggered — now allocate all blocks at once
  4. ext4 finds one large contiguous region = one extent!
  5. Perfectly contiguous file, zero fragmentation
⚠️

The delayed allocation gotcha: Because block allocation is deferred, disk space might be overcommitted. You can write a 10GB file to a 5GB filesystem and only get an error at flush time, not at write time. This surprised many applications. Handle ENOSPC (no space) at write time AND at close/sync time.

Evolution & History

How Linux filesystems evolved from simple to sophisticated

1969
Unix Filesystem (UFS) — The Original
Bell Labs created the first hierarchical filesystem. Everything as a file. Single root /. Inodes introduced. This became the blueprint for everything that followed.
1991
MINIX FS — Linux's Humble Beginning
Linus Torvalds' first Linux used MINIX filesystem — inherited from the educational OS. Limited to 64MB partitions and 14-char filenames. Too restrictive to be practical.
1992
ext — First Extended Filesystem
Linux's first native filesystem. Supported 2GB files, 255-char filenames. No journaling. If power died during write — corruption. Quickly replaced.
1993
ext2 — Workhorse of the 90s
Major upgrade — supports up to 32TB volumes, larger files. Used for decades. Still used for flash drives and boot partitions today because it has no journaling overhead.
1999
ext3 — Journaling Arrives
Ext2 + journaling. A "journal" keeps a log of what's about to change — so if power dies, the journal replays to fix corruption. Backward compatible with ext2. The standard for the 2000s.
2001
XFS — High Performance
Originally from SGI (Irix). Designed for massive files and parallel I/O. Still the default on RHEL/CentOS. Handles terabytes gracefully. Cannot shrink — only grow.
2008
ext4 — Modern Default
Current standard for most Linux. Supports 1EB volumes, 16TB files, nanosecond timestamps, extents (replaces indirect blocks), delayed allocation. Used by Ubuntu, Debian, and more.
2009
btrfs — The Future (Almost)
Copy-on-Write, built-in RAID, snapshots, compression, checksums. Oracle-designed. Default on Fedora, openSUSE. Still maturing — RAID 5/6 considered unstable.
2013+
ZFS on Linux / overlayfs / APFS
ZFS ported to Linux via OpenZFS project. overlayfs enables Docker layers. The ecosystem keeps expanding for specialized use cases.

btrfs, XFS, ZFS — Modern Powerhouses

Next-generation features: CoW, checksums, snapshots, RAID

btrfs
XFS
ZFS
🌲

btrfs = B-tree filesystem. Designed to be the "next ext4". Copy-on-Write means never overwriting data in place — changes create new copies. Makes snapshots near-instant and free.

📸 Snapshots

Instant, space-efficient. Snapshot = new pointer to same data blocks. Only diverging changes consume space. Rollback a broken system in seconds.

🗜️ Inline Compression

Transparent LZO, ZLIB, or ZSTD compression. Files compressed on-the-fly. Can actually make SSDs faster — less data to write.

✅ Checksums

Every data and metadata block has a checksum. Silent corruption (bit rot) detected automatically. ZFS also does this — ext4 does NOT.

📊 Subvolumes

Like virtual partitions inside one filesystem. Each can have its own quota, mount options, and snapshot schedule. Fedora uses them for @ and @home.

⚡

XFS = High-performance journaling filesystem. Built for speed with massive files. Parallel I/O via allocation groups. Default on RHEL/CentOS/Rocky Linux. Cannot shrink — plan ahead!

🏎️ Speed

Allocation groups allow parallel writes. Excellent for databases, video editing, large sequential workloads. Scales to 8EB volumes.

⚠️ Cannot Shrink

XFS can only grow. If you need to resize down, you must backup, reformat, restore. Plan your partition sizes carefully!

🏰

ZFS = The Cadillac of filesystems. Combines filesystem + volume manager. End-to-end checksums, self-healing, built-in RAID (RAIDZ), ARC cache, deduplication. Needs more RAM (1GB min, 8GB+ recommended).

🔧 zpools

ZFS storage pool — one or more disks. Add mirrors, RAIDZ. All filesystems inside share the pool's space.

📸 Snapshots + Send

Snapshots are instant. zfs send streams snapshots over SSH — perfect for incremental remote backups.

🧠 ARC Cache

Adaptive Replacement Cache uses RAM to cache hot data. Dramatically speeds up reads. L2ARC extends to SSD.

Inodes & Data Blocks

The atomic unit of Linux file storage

🎯

Key insight: A filename is NOT part of the file. The filename lives in a directory. The actual file is an inode. This is why hard links work — two names, one inode, one set of data. Delete one name, data stays until ALL names are gone.

What's Inside an Inode?

struct ext4_inode {
mode // permissions + file type
uid, gid // owner user + group
size // file size in bytes
atime // last access time
mtime // last modification
ctime // last status change
links_count // number of hard links
blocks // pointers to data
}
⚠️

NOT in inode: The filename! Filenames are in directory entries. An inode doesn't know its own name.

💡

Inode exhaustion: You can run out of inodes before running out of disk space! Happens with millions of tiny files. Check with df -i

Hard Links vs Soft (Symbolic) Links

🔗 Hard Link

Another directory entry pointing to the same inode. Same file, two names. Delete either name — file survives. Cannot cross filesystem boundaries. Cannot link directories.

ln original.txt hardlink.txt # Both point to same inode ls -li 1048576 -rw-r--r-- 2 user user 100 file.txt 1048576 -rw-r--r-- 2 user user 100 hardlink.txt # Same inode number! ^^

🔀 Symbolic (Soft) Link

A file that contains a path to another file. Like a Windows shortcut. Can cross filesystems. Can link directories. If target deleted → broken link.

ln -s /etc/nginx/nginx.conf nginx.conf ls -la lrwxrwxrwx 1 user user 22 nginx.conf -> /etc/nginx/nginx.conf # l = symlink, -> shows target

Special Bits — The Hidden Power

s SUID Bit

Set on executables: runs with owner's permissions regardless of who runs it. Example: /usr/bin/passwd needs root to change /etc/shadow — SUID makes it work for normal users.

chmod 4755 file → -rwsr-xr-x

s SGID Bit

On files: runs with group's permissions. On directories: new files inherit the directory's group — perfect for shared project dirs.

chmod 2755 dir → drwxr-sr-x

t Sticky Bit

On directories: only the file owner (or root) can delete their own files. Classic example: /tmp — everyone writes, nobody deletes others' files.

chmod 1777 /tmp → drwxrwxrwt

Journaling & Copy-on-Write

How modern filesystems survive power failures

📔 Journaling (ext3/4, xfs)

Before making changes, the filesystem logs the intent in a journal. If power dies mid-write, on next boot it replays/rolls back the journal. Fast recovery, no full fsck needed.

📊

ext4 has 3 journal modes:
journal — safest, slowest (journals data+metadata)
ordered — default, good balance
writeback — fastest, metadata only

🔄 Copy-on-Write (btrfs/zfs)

Never overwrite data. Changes are written to new locations, then the pointer is atomically updated. Old data stays until freed. Power can die at ANY point — always consistent. Makes snapshots trivially cheap.

Before: [A] → block 100
Write: new data → block 200
After: [A] → block 200 ✓
block 100 freed later

RAID — Redundant Array of Independent Disks

Performance + redundancy through disk arrays

RAID 0 — Striping

Data split across 2+ disks. Double the speed. Zero redundancy — one disk fails, all data gone. For scratch disks, video editing where speed > safety.

Disk 1: [A1][B1][C1]
Disk 2: [A2][B2][C2]
Capacity: 2x | Speed: 2x

RAID 1 — Mirroring

Exact copy on 2+ disks. Survive one disk failure. Read speed improved. Write speed = single disk. For OS drives, critical data.

Disk 1: [A][B][C]
Disk 2: [A][B][C] (copy)
Capacity: 1x | Survive: 1 failure

RAID 5 — Striping + Parity

3+ disks. Data + parity striped across all. One disk failure tolerated — rebuild from parity. 66%+ efficiency. Popular for NAS.

Disk 1: [A1][B1][Cp]
Disk 2: [A2][Bp][C1]
Disk 3: [Ap][B2][C2]
Capacity: (N-1)x

RAID 10 — Mirror + Stripe

4+ disks. Mirror pairs, then stripe across pairs. Best of both worlds. Multiple disk failures tolerated (as long as not same pair). Enterprise standard.

[Mirror1: D1+D2] + [Mirror2: D3+D4]
striped across
Capacity: 2x | Survive: 2 failures

LVM Internals — How Thin Provisioning & Snapshots Work

Beyond basic LVM — the internals that every senior engineer must know

Basic LVM (Physical Volumes → Volume Groups → Logical Volumes) is covered in every tutorial. But senior engineers need to understand: How does a 1TB thin-provisioned LV exist on a 500GB disk? How does an LVM snapshot avoid copying 100GB of data instantly? What happens at the PE (Physical Extent) level? This section goes deep.

Physical Extents — The Atomic Unit of LVM

LVM divides all Physical Volumes into fixed-size chunks called Physical Extents (PEs). Default size: 4MB. A 100GB disk = 25,600 PEs. The Volume Group is a pool of these PEs. A Logical Volume = a mapping of Logical Extents (LEs) → Physical Extents.

LVM Striping — Spread I/O Across Disks

LVM can stripe a logical volume across multiple physical volumes, similar to RAID 0. Writes alternate between PVs in stripe-size chunks. A 2-disk stripe doubles sequential throughput.

💡

RAID — The Real Math & When Redundancy Fails

XOR parity, RAIDZ differences, URE rates, and why RAID is not a backup

How RAID 5 Parity Actually Works — XOR Math

RAID 5 parity is based on XOR (exclusive OR). For any set of bits A, B: P = A XOR B. If A is lost: A = P XOR B. This extends to n disks: P = D1 XOR D2 XOR D3. Lose any one: reconstruct from XOR of the others.

/* RAID 5 example with 3 disks + 1 parity */
Disk 1: 1010 1100 (data)
Disk 2: 0110 1001 (data)
Disk 3: 1100 0011 (data)
Parity: 0100 0110 (P = D1 XOR D2 XOR D3)
/* Disk 2 dies. Rebuild: */
D2 = D1 XOR D3 XOR P
D2 = 1010 1100 XOR 1100 0011 XOR 0100 0110
D2 = 0110 1001 ✓ Correct!

RAID 6 — Double Parity (P + Q)

RAID 6 adds a second parity symbol Q using Galois Field arithmetic (GF(2^8)). P = simple XOR. Q = XOR with coefficients from GF. This allows reconstruction of ANY two lost disks. The math is more complex but same principle — algebra over finite fields.

⚠️

RAID 5 vulnerability window: During a RAID 5 rebuild (which takes hours on large disks), if a SECOND drive fails, you lose everything. With today's 8TB+ drives, rebuild takes 24+ hours. The probability of a second failure during rebuild is non-trivial. Use RAID 6 for any data you care about. Or better: ZFS RAIDZ2.

URE — Why RAID 5 on Large Disks is Dangerous

Unrecoverable Read Error (URE): Modern HDDs have a spec of 1 URE per 10^14 bits read (10^14 bits = 12.5TB). During RAID 5 rebuild on a failed 8TB disk, you must read all remaining data = ~16TB on a 3-disk RAID 5. Probability of hitting a URE during rebuild: 1 - (1 - 1/10^14)^(16×8×10^12) ≈ 16%. One in six rebuilds fails!

Don't use for critical data
RAID 5 on large HDDs, Software RAID on consumer drives without regular scrubs, RAID 5 for database write-heavy workloads
Use instead
ZFS RAIDZ2 (double parity + checksums to catch UREs), RAID 10 (better rebuild performance), Regular btrfs scrubs to detect bit rot proactively

NFS, Ceph & Distributed Filesystem Internals

How files cross network boundaries — and the CAP theorem tradeoffs

NFS v3 vs v4 — The Critical Differences

NFS v3 — Stateless

Each RPC call is self-contained. Server remembers nothing between calls. If server reboots, clients retry and continue. Simple, robust. But: no proper file locking (NLM is a separate protocol), no strong consistency guarantees, AUTH_SYS only (uid/gid — easy to spoof).

NFS v4 — Stateful

Single TCP connection. Server tracks client state (open files, locks). Real file locking built-in. Kerberos authentication (krb5). Compound RPCs (multiple operations in one network round trip). ACLs. Significantly better security and performance. Default in modern Linux.

Ceph — The Architecture Behind Cloud Storage

Ceph is what powers OpenStack, Kubernetes persistent volumes, and many cloud storage backends. Its genius is the CRUSH algorithm — Controlled Replication Under Scalable Hashing. Instead of a lookup table (which has a single point of failure), CRUSH is a deterministic algorithm: given an object name and the cluster map, any node can independently calculate where that object lives. No central metadata server for data placement.

LUKS Encryption — The Internals

LUKS (Linux Unified Key Setup) is the standard for disk encryption. Understanding its design explains why it's secure and flexible:

/* LUKS header layout on disk */
Bytes 0-511: LUKS magic + version + cipher info
Bytes 512-575: Master Key UUID
Bytes 576+: Key Slot 0 (password 1 → key material)
Key Slot 1 (password 2 → key material)
... up to 8 key slots
After header: Encrypted data (using Master Key)
/* Each key slot contains the Master Key encrypted with PBKDF2(password). Change password = re-encrypt one slot. Master Key never changes. Wipe a slot = revoke that password without re-encrypting data. */
bash — LUKS advanced operations
# Create LUKS2 with strong parameters cryptsetup luksFormat --type luks2 \ --cipher aes-xts-plain64 \ --key-size 512 \ --hash sha256 \ --iter-time 5000 \ /dev/sdb # Add a second password (e.g., recovery key) cryptsetup luksAddKey /dev/sdb # prompts for new key # Add a keyfile (for auto-unlock on boot) dd if=/dev/urandom of=/root/keyfile bs=512 count=8 cryptsetup luksAddKey /dev/sdb /root/keyfile # In /etc/crypttab: data_crypt /dev/sdb /root/keyfile luks # Backup LUKS header (CRITICAL — if header corrupted, data is gone) cryptsetup luksHeaderBackup /dev/sdb --header-backup-file /backup/sdb-luks-header.bin # Inspect LUKS header cryptsetup luksDump /dev/sdb # Wipe a key slot (revoke a password) cryptsetup luksKillSlot /dev/sdb 1 # wipe slot 1

❓ "Disk is not full but I can't create new files. Why?"

▶
Answer: Inode exhaustion. The filesystem has run out of inodes — the metadata structures representing files. Each file consumes exactly one inode, regardless of size. If you have millions of tiny files (log files, email queues, cache entries, npm node_modules), you can exhaust inodes while disk space appears available.
df -i # check inode usage Filesystem Inodes IUsed IFree IUse% Mounted on /dev/sda1 6553600 6553600 0 100% / ← FULL! # Fix: find which directory has millions of files find / -xdev -printf '%h\n' | sort | uniq -c | sort -rn | head # Prevention: format with more inodes or use large inodes/extents on modern FS mkfs.ext4 -N 20000000 /dev/sdb # 20M inodes explicitly

❓ "What happens when you delete a file that's open by a process?"

▶
Answer: The directory entry is removed immediately (the filename disappears from ls), but the inode and data blocks are NOT freed until the link count reaches zero AND all file descriptors referencing it are closed. While the process keeps the file open, the data is still accessible. The space is not reclaimed until the last fd closes.

This is why: log rotation works (old log deleted, new one created, logger continues writing to old inode via open fd). And why a deleted file can still consume disk space if a process holds it open. Diagnose with: lsof | grep deleted
lsof | grep deleted nginx 1234 www 4w REG 8,1 2147483648 /var/log/nginx/access.log (deleted) # 2GB "deleted" file still consuming space — nginx fd still open # Fix: send SIGHUP to nginx to reopen log files, then space reclaimed kill -HUP $(pgrep nginx)

❓ "Explain journaling modes and when you'd use each"

▶
data=journal: Everything journaled. Maximum safety. Use for: critical configuration files, when data loss is absolutely unacceptable, testing. Rarely used in production due to 2× write overhead.

data=ordered (default): Metadata journaled, data flushed to disk before metadata commit. Good safety/performance balance. Use for: general purpose filesystems, most production servers.

data=writeback: Only metadata journaled. Fastest. Use for: database servers (where the database has its own durability via WAL/redo log), scratch/temp filesystems, when a battery-backed RAID controller handles durability guarantees.

❓ "Why is btrfs faster on SSDs but can be slower on HDDs?"

▶
On SSDs: CoW writes new data to free locations — SSDs prefer this (no erase-before-write penalty). B-tree metadata updates are random writes — SSDs handle random I/O at nearly the same speed as sequential. Compression reduces writes, extending SSD life. No seek penalty means B-tree traversal is fast.

On HDDs: CoW's random writes are catastrophic for HDDs — each new block location may require a seek. B-tree metadata scattered across disk. Fragmentation builds up faster. For spinning disks, ext4's eager contiguous allocation strategy can outperform CoW significantly for sequential workloads.

Mount option fix: mount -o autodefrag helps on HDDs by defragmenting files on access. And for database-style workloads on HDDs, chattr +C disables CoW for individual files/directories.
