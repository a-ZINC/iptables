Hands-on
the building analogy — remember this throughout
💡 Core mental model

Your Linux machine is a building. Every packet is a person visiting. The building has three main corridors: one for visitors arriving at your building (INPUT), one for visitors you send out (OUTPUT), and one for visitors just passing through to reach somewhere else (FORWARD).

Along each corridor are checkpoints (tables) with guards following your rulebook (rules). Guards can: let you through (ACCEPT), throw you out silently (DROP), send you back with a message (REJECT), change your destination sticker (DNAT), or change your name badge (SNAT).

📦 What is a Packet?

Every network message is broken into packets. Each has a header (src IP, dst IP, ports, protocol, TTL) and a payload (the actual data). iptables only reads headers — it never inspects payloads.

🔗 What is a Chain?

A chain is a list of rules checked top-to-bottom. The first matching rule wins. If no rule matches, the chain's default policy applies (ACCEPT or DROP). There are 5 built-in chains, each at a different point in the packet's journey.

📋 What is a Table?

Tables group rules by purpose. raw = pre-screening, mangle = modifying headers, nat = translating addresses, filter = allowing or blocking. Tables run in a fixed order: raw → mangle → nat → filter.

rule anatomy

⚖️ Anatomy of a Rule

Every iptables rule = match conditions + action (target). Conditions can match on protocol, ports, IPs, interfaces, connection state, and more.

rule structure
# iptables [table] [command] [chain] [match] -j [target]
iptables -t filter -A INPUT   # append to INPUT chain
         -p tcp               # match: TCP protocol
         --dport 22           # match: destination port 22
         -s 10.0.0.5          # match: source IP
         -j ACCEPT            # action: accept it

# -t filter is the default — this is equivalent:
iptables -A INPUT -p tcp --dport 22 -s 10.0.0.5 -j ACCEPT

Rules are checked top to bottom. The first matching rule wins. If no rule matches → default policy (ACCEPT or DROP).

🛠️ Managing Rules

commands
# View rules with counters and line numbers
iptables -L -n -v --line-numbers

# Append rule to end of chain
iptables -A INPUT -p tcp --dport 80 -j ACCEPT

# Insert rule at position 1 (top)
iptables -I INPUT 1 -p tcp --dport 80 -j ACCEPT

# Delete rule by line number
iptables -D INPUT 3

# Flush (delete) all rules
iptables -F

# Set default policy
iptables -P INPUT DROP

# Save rules (persist after reboot)
iptables-save > /etc/iptables/rules.v4
targets — what to do when a rule matches

✅ ACCEPT

Let the packet through. Processing stops — no more rules checked in this chain.

-j ACCEPT

🚫 DROP

Silently discard. Sender gets no response — connection times out (~30s). Best for public services; attackers can't confirm the host exists.

-j DROP

⛔ REJECT

Discard + send ICMP unreachable back. Sender fails immediately. Better for internal networks where fast feedback matters.

-j REJECT
-j REJECT --reject-with tcp-reset

📝 LOG

Non-terminating — writes to syslog then continues to next rule. Always add LOG before DROP to see what you're blocking.

-j LOG --log-prefix "BLOCKED: "
# Read: tail -f /var/log/syslog | grep BLOCKED

🏷️ MARK

Tag packet with a number — kernel-internal only (not visible on wire). Policy routing (ip rule) and tc can read this tag and act on it.

-j MARK --set-mark 1

🚦 NOTRACK

Skip connection tracking for this packet (raw table only). Saves memory for high-volume flows like DNS where you don't need stateful rules.

-t raw ... -j NOTRACK
⚠️ DROP vs REJECT
DROP: silent discard, port scanners see "filtered", sender waits ~30s to timeout. Use for public-facing services.  |  REJECT: sends ICMP back, sender fails immediately, port scanners see "closed". Use inside trusted networks where fast failure is helpful.

The 4 Tables

Each table has exactly one job — and they run in a fixed order

💡 Tables = specialist departments

raw = pre-screener (earliest intercept, before conntrack), mangle = packet editor (modify headers, set marks), nat = address translator (DNAT/SNAT), filter = gatekeeper (ACCEPT/DROP). The order is always raw → mangle → nat → filter — hardcoded in the kernel, cannot be changed.

🟢 raw — The Pre-Screener

Runs first, before connection tracking, before NAT. Available in PREROUTING and OUTPUT only.

Primary use: NOTRACK — skip conntrack for high-volume flows. Critical for DNS servers, game servers, streaming. Without NOTRACK, conntrack table fills up and new connections are silently dropped.

raw table examples
# Skip conntrack for DNS (saves memory)
iptables -t raw -A PREROUTING -p udp --dport 53 -j NOTRACK
iptables -t raw -A OUTPUT -p udp --sport 53 -j NOTRACK

# Early drop before conntrack wastes resources
iptables -t raw -A PREROUTING -s 1.2.3.4 -j DROP

# Log in raw state (before any modification)
iptables -t raw -A PREROUTING -p tcp --dport 80 -j LOG --log-prefix "RAW: "

🟣 mangle — The Packet Editor

Modifies packet header fields — does NOT block or redirect. Available in all 5 chains.

Use for: MARK (tag for policy routing), TTL (change hop count), DSCP/TOS (QoS priority bits), TCPMSS (VPN MTU fix — prevents fragmentation).

mangle table examples
# Tag HTTP for policy routing (to use VPN interface)
iptables -t mangle -A PREROUTING -p tcp --dport 80 -j MARK --set-mark 1

# MSS clamp — CRITICAL for VPN tunnels (prevents stalling on large files)
iptables -t mangle -A FORWARD -o tun0 \
  -p tcp --tcp-flags SYN,RST SYN -j TCPMSS --clamp-mss-to-pmtu

# Set DSCP for VoIP quality
iptables -t mangle -A OUTPUT -p udp --dport 5060 -j DSCP --set-dscp-class EF

🟡 nat — The Address Changer

Rewrites source or destination IPs/ports. Only runs on the first packet of each connection — conntrack handles all subsequent packets automatically.

DNAT in PREROUTING (port forwarding) · SNAT/MASQUERADE in POSTROUTING (internet sharing) · REDIRECT (transparent proxy to self)

nat table examples
# Port forward :80 → internal server
iptables -t nat -A PREROUTING -p tcp --dport 80 -j DNAT --to 192.168.1.10:80

# Share internet (MASQUERADE — dynamic IP)
iptables -t nat -A POSTROUTING -o eth0 -j MASQUERADE

# Transparent proxy — redirect HTTP to local proxy
iptables -t nat -A PREROUTING -p tcp --dport 80 -j REDIRECT --to-port 3128

🔵 filter — The Gatekeeper

The main firewall table. ACCEPT or DROP packets here. This is the default table — iptables -A INPUT without -t goes here. Available in INPUT, FORWARD, OUTPUT only.

95% of all firewall rules live in the filter table.

filter table examples
# The essential stateful firewall trio:
iptables -A INPUT -i lo -j ACCEPT
iptables -A INPUT -m state --state ESTABLISHED,RELATED -j ACCEPT
iptables -A INPUT -m state --state INVALID -j DROP

# Then add specific services:
iptables -A INPUT -p tcp --dport 22 -j ACCEPT
iptables -A INPUT -p tcp -m multiport --dports 80,443 -j ACCEPT

# Set default deny LAST:
iptables -P INPUT DROP

The 5 Chains

Checkpoints along the packet's journey — each runs at a specific moment

PREROUTING — "Arrival Gate"

The very first stop for all incoming packets — before the kernel decides where to route them. Tables: raw → mangle → nat.

⚡ Key power: DNAT here changes the destination IP before the routing decision. So if you DNAT to another host's IP, routing sends it to FORWARD (not INPUT).

Use: port forwarding, transparent proxies, NOTRACK for high-speed flows.

INPUT — "Receiving Desk"

For packets destined for this machine — a local process (nginx, sshd, your app) will receive them. Tables: mangle → filter.

Main firewall chain: "allow SSH port 22", "allow HTTP port 80", "drop everything else". Does NOT run for forwarded packets.

Use: service access control, ingress firewall.

FORWARD — "Transit Lounge"

For packets passing THROUGH this machine to a different host entirely. Tables: mangle → filter.

⚠️ Requires net.ipv4.ip_forward=1 — without it, the kernel drops forwarded packets before they even reach this chain. All Docker, VPN, and VM traffic goes through FORWARD on the host.

Most common mistake: DNAT rule set, FORWARD rule missing → packets silently dropped.

OUTPUT — "Departure Gate"

For packets generated by this machine — curl, ping, your application. Tables: raw → mangle → nat → filter.

Locally-generated packets completely skip PREROUTING — their journey starts here. You can DNAT in OUTPUT to redirect your own machine's outbound traffic to a local proxy.

POSTROUTING — "Final Check"

The very last chain — just before a packet leaves the machine, after the routing decision picks the outgoing interface. Tables: mangle → nat.

Both forwarded AND locally-generated packets pass through here. SNAT and MASQUERADE live here — source IP is rewritten just before the packet hits the wire. MSS clamping for VPN tunnels also goes here (because you know the exit interface at this point).

✅ The critical interaction: DNAT changes routing
When DNAT in PREROUTING rewrites the destination IP, the routing decision runs on the new IP. If the new IP belongs to another host on your LAN — routing sends the packet to FORWARD, not INPUT. This is why every DNAT port-forward rule must be paired with a FORWARD ACCEPT rule.

NAT Deep Dive

DNAT, SNAT, MASQUERADE, REDIRECT — how address translation works

🎯 DNAT — Change Destination

Rewrites where the packet is going. Used in PREROUTING (for incoming traffic) or OUTPUT (for your own traffic). Classic use: port forwarding.

After DNAT, routing uses the new destination. If it's another host → FORWARD chain is used. Always add a FORWARD rule alongside DNAT!

DNAT examples
# External :80 → internal server 192.168.1.10
iptables -t nat -A PREROUTING \
  -p tcp --dport 80 \
  -j DNAT --to-destination 192.168.1.10:80

# !! MUST also add FORWARD rule:
iptables -A FORWARD -p tcp \
  -d 192.168.1.10 --dport 80 -j ACCEPT

# Port forward with port change: :8080 → internal :80
iptables -t nat -A PREROUTING \
  -p tcp --dport 8080 \
  -j DNAT --to-destination 192.168.1.10:80

# Transparent proxy (REDIRECT = DNAT to self)
iptables -t nat -A PREROUTING -p tcp --dport 80 \
  -j REDIRECT --to-port 3128

📤 SNAT / MASQUERADE — Change Source

Rewrites where the packet appears to come from. Used in POSTROUTING. Used so private hosts can share a public IP to reach the internet.

SNAT: specify a static source IP (slightly faster). MASQUERADE: dynamically uses the outgoing interface's current IP (handles DHCP/cloud VMs where IP changes).

SNAT/MASQUERADE
# MASQUERADE — dynamic IP (home router, cloud VM)
iptables -t nat -A POSTROUTING -o eth0 -j MASQUERADE

# SNAT — static public IP (dedicated server, faster)
iptables -t nat -A POSTROUTING \
  -s 192.168.0.0/24 -o eth0 \
  -j SNAT --to-source 203.0.113.1

# SNAT to a pool of IPs (load balance outbound NAT)
iptables -t nat -A POSTROUTING \
  -j SNAT --to-source 203.0.113.1-203.0.113.5
⚠️ The #1 NAT Gotcha — Port Forwarding
The problem: You add DNAT, test it, and traffic never arrives at the backend. Why: DNAT changes destination to 192.168.1.10. Routing says "192.168.1.10 is another host → FORWARD". FORWARD default policy = DROP. Packet dies silently. The fix: iptables -A FORWARD -p tcp -d 192.168.1.10 --dport 80 -j ACCEPT. Without this, your DNAT "works" but packets are dropped before reaching the backend.
nat + conntrack = automatic reverse translation

🔁 How NAT + Conntrack Work Together

NAT only runs on the first packet of each connection. After that, conntrack records the translation. When the reply arrives, conntrack automatically reverses the translation — without hitting the NAT chain again.

Example: Internal 192.168.1.5 makes request → MASQUERADE changes source to 203.0.113.1 → internet server responds to 203.0.113.1 → conntrack knows 192.168.1.5 made that request → automatically delivers reply to 192.168.1.5. You write zero rules for this. It just works.

Connection Tracking

The kernel's memory of active connections — what makes stateful firewalling possible

💡 What conntrack does

Conntrack is a kernel subsystem that keeps a table of every active connection. When you SSH into a server, conntrack writes an entry: src:192.168.1.5:54123 → dst:10.0.0.1:22, state:ESTABLISHED. When the reply comes back, conntrack matches it and marks it ESTABLISHED. Your firewall rule -m state --state ESTABLISHED -j ACCEPT then lets it through — without any per-service "allow return traffic" rule.

📊 The 4 Connection States

StateMeaning
NEWFirst packet. No existing entry. TCP SYN, first UDP.
ESTABLISHEDActive connection. Both sides exchanged packets.
RELATEDNew connection related to existing (FTP data, ICMP errors).
INVALIDDoesn't match known connection. Wrong TCP flags. Always DROP these.

🛡️ The Stateful Firewall Trio

These three rules are the foundation of every Linux server firewall. Order matters.

essential rules
# 1. Always allow loopback (apps → 127.0.0.1)
iptables -A INPUT -i lo -j ACCEPT

# 2. Allow reply traffic — the stateful rule
iptables -A INPUT -m state \
  --state ESTABLISHED,RELATED -j ACCEPT

# 3. Drop malformed packets
iptables -A INPUT -m state \
  --state INVALID -j DROP

# Then add service rules, then set policy LAST:
iptables -P INPUT DROP

📋 Conntrack Commands & Limits

conntrack management
# View all tracked connections
conntrack -L

# Watch new connections in real time

iptables -t raw -A PREROUTING -p udp --dport 53 -j NOTRACK
iptables -t raw -A OUTPUT -p udp --sport 53 -j NOTRACK

Routing Fundamentals

How the kernel decides where every packet goes

💡 The postal worker analogy

You are a postal worker with a stack of letters (packets). Each letter has a destination address. You have a routing table on the wall: "Letters for 192.168.1.x → tray A (eth1)", "Letters for 10.0.0.x → tray B (eth2)", "Everything else → main post office (default gateway at 192.168.1.1)".

The rule: most specific address wins. 192.168.1.0/24 beats 0.0.0.0/0. A /32 (single host) is the most specific possible. If no route matches at all — letter is returned "undeliverable" (ICMP network unreachable).

❓ What Routing Answers

One question: "Given this packet's destination IP — which interface do I send it out of, and what is the next hop (gateway)?"

Every packet triggers this lookup: inbound after PREROUTING (possibly modified by DNAT), outbound from local processes, and forwarded packets.

📍 Longest Prefix Match

When multiple routes match a destination, the most specific wins — the longest subnet prefix (highest number after the slash).

10.0.1.0/24 beats 10.0.0.0/8 for destination 10.0.1.5. A /32 (single-host route) always wins against any broader prefix.

🚪 Default Route

The catch-all: 0.0.0.0/0. Matches everything not matched by a more specific route. Points to your default gateway — how packets reach the internet.

Without a default route, packets to unknown destinations fail with "Network unreachable".

the routing table
$ ip route showmain routing table

essential routing commands
# View routing table
ip route show
default via 192.168.1.1 dev eth0 proto dhcp metric 100
192.168.1.0/24 dev eth0 proto kernel scope link src 192.168.1.50

# Find which route wins for a specific destination
ip route get 8.8.8.8
8.8.8.8 via 192.168.1.1 dev eth0 src 192.168.1.50

# Add a static route
ip route add 10.0.5.0/24 via 192.168.1.254

# Add route via specific interface (directly connected)
ip route add 10.0.5.0/24 dev eth1

# Add/replace default gateway
ip route add default via 192.168.1.1

# Delete a route
ip route del 10.0.5.0/24

# Enable IP forwarding (required to act as router)
echo 1 > /proc/sys/net/ipv4/ip_forward
# Persistent (survives reboot):
echo "net.ipv4.ip_forward=1" >> /etc/sysctl.conf && sysctl -p

🗺️ ARP — IP to MAC Address

Routing tells you which interface and which gateway. But Ethernet needs MAC addresses. ARP resolves the gateway's IP to its MAC so the frame can actually be sent.

arp commands
# View ARP cache (IP → MAC mappings)
ip neigh show
192.168.1.1 dev eth0 lladdr aa:bb:cc:dd:ee:ff REACHABLE
192.168.1.10 dev eth0 lladdr 11:22:33:44:55:66 STALE

# ARP states:
#  REACHABLE  = recently confirmed valid
#  STALE      = will re-verify on next use
#  FAILED     = ARP sent, no response (host unreachable)

# Force re-resolution (useful after gateway IP changes)
ip neigh del 192.168.1.1 dev eth0

# Add static ARP entry (prevents ARP spoofing)
ip neigh add 192.168.1.1 lladdr aa:bb:cc:dd:ee:ff dev eth0 nud permanent
routing diagnostics flowchart
systematic debug approach
# 1. Can you reach anything at all?
ping -c3 127.0.0.1      # loopback — tests local stack
ping -c3 192.168.1.1    # gateway — tests local network

Policy-Based Routing

Route packets differently based on source, mark, or interface — not just destination

💡 Why standard routing isn't enough

Standard routing only looks at the destination IP. But real-world needs are more complex: "Traffic from eth1 must reply via eth1's ISP" or "HTTP/HTTPS goes via VPN, everything else uses the normal internet." Policy-Based Routing (PBR) solves this by maintaining multiple routing tables and rules for which table to consult.

📋 ip rule — The Policy Database

Before the main routing table, the kernel checks ip rule — a list of policies in priority order. Each says: "if packet matches X, use routing table Y."

ip rule management
ip rule show
0:      from all lookup local     ← highest priority
32766:  from all lookup main      ← normal routing table
32767:  from all lookup default   ← fallback (empty)

# Route by source IP (lower number = higher priority)
ip rule add from 192.168.2.0/24 table 100 priority 100

# Route by iptables MARK
ip rule add fwmark 1 table 200 priority 200

# Route by incoming interface
ip rule add iif eth1 table 100 priority 150

# Remove a rule
ip rule del from 192.168.2.0/24 table 100

🔀 MARK + ip rule — Most Powerful Pattern

Tag packets with iptables MARK in the mangle table, then route tagged packets via a different routing table with ip rule. Combines firewall logic with routing decisions.

route HTTP via VPN, rest via eth0
# 1. Tag HTTP/HTTPS with mark=1
iptables -t mangle -A OUTPUT -p tcp --dport 80 -j MARK --set-mark 1
iptables -t mangle -A OUTPUT -p tcp --dport 443 -j MARK --set-mark 1

# 2. Create a named routing table
echo "50 vpn" >> /etc/iproute2/rt_tables
ip route add default dev tun0 table vpn

# 3. Route marked=1 packets via VPN table
ip rule add fwmark 1 table vpn priority 50

# Verify
ip route get 8.8.8.8          # normal → eth0
ip route get 8.8.8.8 mark 1   # marked → tun0

🌐 Dual-ISP Setup — Fixing Asymmetric Routing

Server with two ISPs. Without PBR: connections coming in via ISP2 reply via ISP1's gateway → ISP1 drops them (wrong source IP). PBR ensures replies use the same ISP the connection arrived on.

dual-ISP policy routing
# eth0 = ISP1 (IP: 1.2.3.4, gateway: 1.2.3.1)
# eth1 = ISP2 (IP: 5.6.7.8, gateway: 5.6.7.1)

# 1. Name two routing tables
echo "100 isp1" >> /etc/iproute2/rt_tables
echo "200 isp2" >> /etc/iproute2/rt_tables

# 2. Populate ISP1 table — default via ISP1
ip route add default via 1.2.3.1 dev eth0 table isp1
ip route add 1.2.3.0/30 dev eth0 scope link table isp1

# 3. Populate ISP2 table — default via ISP2
ip route add default via 5.6.7.1 dev eth1 table isp2
ip route add 5.6.7.0/30 dev eth1 scope link table isp2

# 4. Policy: replies from ISP1 IP → use ISP1 table
ip rule add from 1.2.3.4 table isp1 priority 100
ip rule add from 5.6.7.8 table isp2 priority 200

# Verify
ip route get 8.8.8.8 from 1.2.3.4    # → via ISP1
ip route get 8.8.8.8 from 5.6.7.8    # → via ISP2

⚡ DNAT Changes the Routing Decision

When DNAT in PREROUTING rewrites the destination IP, the routing lookup runs on the new IP.

Original dst: 203.0.113.1 (this machine's IP) → routing → INPUT
After DNAT: dst: 192.168.1.10 (another host) → routing → FORWARD

This is why DNAT to another host always needs a FORWARD ACCEPT rule.

🏷️ MARK Bridges Both Worlds

The most powerful pattern: iptables tags packets in mangle → ip rule reads the mark and picks a routing table → different traffic flows exit via different interfaces.

VoIP tagged MARK=1 → routed via low-latency ISP table. HTTP tagged MARK=2 → routed via VPN. Everything else → normal default route.

🔑 The 5 key relationships
1. Routing decides the path — iptables rules only fire if the packet reaches that chain.  2. DNAT changes routing — DNAT in PREROUTING can redirect to FORWARD instead of INPUT.  3. conntrack enables stateful rules — ESTABLISHED matches rely on conntrack entries.  4. MARK bridges iptables and routing — tag in mangle, route by tag with ip rule.  5. ip_forward enables FORWARD — without it, forwarded packets die before iptables sees them.

Project 1: Stateful Firewall

Build a production-grade default-deny firewall from scratch

Project 01 of 06
Production Stateful Firewall
Configure a default-deny INPUT firewall with SSH, HTTP, HTTPS, and stateful connection tracking. Learn the exact rule ordering that keeps your SSH session alive while hardening the server. This setup runs on every production Linux server.
Beginner~15 minfilter table · INPUT chain

Steps — run in order

1

Clean start — flush rules, reset policies to ACCEPT (safe starting point)

sudo iptables -F && sudo iptables -P INPUT ACCEPT # Never set DROP policy before adding ESTABLISHED rule
2

Safety net — if you break SSH, rules auto-restore in 60s

(sleep 60 && sudo iptables -F && sudo iptables -P INPUT ACCEPT) &
3

Allow loopback — apps communicate with each other via 127.0.0.1

sudo iptables -A INPUT -i lo -j ACCEPT
4

Allow established connections — the stateful rule that lets replies through

sudo iptables -A INPUT -m state --state ESTABLISHED,RELATED -j ACCEPT
5

Drop invalid packets — broken/spoofed packets with wrong TCP flags

sudo iptables -A INPUT -m state --state INVALID -j DROP
6

Allow SSH (always add before setting DROP policy!)

sudo iptables -A INPUT -p tcp --dport 22 -j ACCEPT
7

Allow HTTP and HTTPS using multiport match

sudo iptables -A INPUT -p tcp -m multiport --dports 80,443 -j ACCEPT
8

Allow ICMP (ping)

sudo iptables -A INPUT -p icmp -j ACCEPT
9

Set default DROP — now all unmatched traffic is blocked

sudo iptables -P INPUT DROP # Do this LAST — your current SSH session should still work

Verify it works

✅ Run these checks
sudo iptables -L INPUT -n -v --line-numbers ping -c2 127.0.0.1 curl -s --max-time 3 http://localhost | head -3 sudo iptables -Z && ping -c1 8.8.8.8 && sudo iptables -L INPUT -v

Cleanup

sudo iptables -F && sudo iptables -P INPUT ACCEPT

What you learn

Rule order is critical — ESTABLISHED before DROP policy, loopback before DROP, specific ACCEPTs before DROP
The ESTABLISHED rule is what makes the firewall "stateful" — allows replies without per-service return rules
Default DROP policy is set LAST to avoid cutting off your SSH session during setup
Zero counters (-Z), send test traffic, check which counter incremented — tells you exactly which rule matched

Project 2: NAT Gateway

Turn this machine into an internet gateway for your LAN

Project 02 of 06
Linux NAT Internet Gateway
Configure MASQUERADE so hosts on your LAN can share this machine's internet connection. Enable ip_forward, set up FORWARD rules, add MASQUERADE. This is exactly how home routers, cloud NAT gateways, and Docker networking work.
Beginner~20 minnat table · FORWARD + POSTROUTING

Steps

1

Enable IP forwarding — without this, kernel drops forwarded packets before iptables

echo 1 | sudo tee /proc/sys/net/ipv4/ip_forward cat /proc/sys/net/ipv4/ip_forward # Must output: 1
2

Find your WAN interface (connected to internet)

ip route show | grep default # Note interface after "dev" — eth0, ens3, etc.
3

Add MASQUERADE — rewrites source IP of outbound packets to this machine's public IP

sudo iptables -t nat -A POSTROUTING -o eth0 -j MASQUERADE # Replace eth0 with your WAN interface
4

Allow forwarding from LAN to WAN (replace eth1 with your LAN interface)

sudo iptables -A FORWARD -i eth1 -o eth0 -j ACCEPT
5

Allow established return traffic (internet replies back to LAN hosts)

sudo iptables -A FORWARD -m state --state ESTABLISHED,RELATED -j ACCEPT
6

Make ip_forward persistent across reboots

echo "net.ipv4.ip_forward=1" | sudo tee -a /etc/sysctl.conf && sudo sysctl -p

Verify it works

✅ On the gateway machine
sudo iptables -t nat -L POSTROUTING -n -v sudo iptables -L FORWARD -n -v cat /proc/sys/net/ipv4/ip_forward
✅ On a LAN host (set gateway to this machine's IP)
curl ifconfig.me # Should show this machine's public IP

Cleanup

sudo iptables -F && sudo iptables -t nat -F && echo 0 | sudo tee /proc/sys/net/ipv4/ip_forward

What you learn

ip_forward MUST be 1 — it's a kernel setting separate from iptables; without it FORWARD chain is never reached
MASQUERADE rewrites source IP in POSTROUTING, after the routing decision picks the exit interface
conntrack handles reverse translation of replies automatically — no rules needed for return traffic
FORWARD ESTABLISHED rule is still needed — for the return journey of packets initiated from LAN

Project 3: Port Forwarding

DNAT external traffic to an internal server — and understand the gotcha

Project 03 of 06
Port Forwarding with DNAT
Forward external port 8080 to an internal nginx on port 80. See DNAT change the destination, understand why FORWARD rules are required, and verify with packet counters. This is how every load balancer, reverse proxy, and home firewall works.
Intermediate~25 minnat + filter tables · PREROUTING + FORWARD

Steps

1

Start nginx on port 80

sudo systemctl start nginx && curl -s http://localhost | head -3
2

Clean start

sudo iptables -F && sudo iptables -t nat -F
3

REDIRECT port 8080 → port 80 (DNAT to self — simplest port forward)

sudo iptables -t nat -A PREROUTING -p tcp --dport 8080 -j REDIRECT --to-port 80
4

Test it — port 8080 should serve nginx

curl -s http://localhost:8080 | head -3
5

Now try DNAT to another host (the gotcha scenario)

sudo iptables -t nat -A PREROUTING -p tcp --dport 9090 -j DNAT --to-destination 192.168.1.10:80 # Replace 192.168.1.10 with another machine's IP
6

Enable forwarding and add the REQUIRED FORWARD rule

echo 1 | sudo tee /proc/sys/net/ipv4/ip_forward sudo iptables -A FORWARD -p tcp -d 192.168.1.10 --dport 80 -j ACCEPT sudo iptables -A FORWARD -m state --state ESTABLISHED,RELATED -j ACCEPT # Without this FORWARD rule, packets are silently dropped!
7

Watch counters to confirm DNAT is firing

sudo iptables -t nat -Z && curl -s http://localhost:8080 > /dev/null && sudo iptables -t nat -L PREROUTING -v

Verify & Debug

✅ Check DNAT is working
sudo iptables -t nat -L PREROUTING -n -v curl -s http://localhost:8080 | head -3 conntrack -L | grep 8080
⚠️ The FORWARD Gotcha
If DNAT to another host (not self) doesn't work: DNAT rewrites destination → routing says "go to FORWARD" → FORWARD default policy = DROP → packet dies silently. Fix: always add iptables -A FORWARD -d [backend] -p tcp --dport [port] -j ACCEPT.

Cleanup

sudo iptables -F && sudo iptables -t nat -F

What you learn

REDIRECT is the simplest DNAT (change destination to self) — no FORWARD rule needed
DNAT to another host changes the routing decision — packet goes to FORWARD, not INPUT
Zero NAT counters and test — counter on PREROUTING rule confirms DNAT is firing
conntrack -L shows the translated connection entry with both original and reply tuples

Project 4: Rate Limiting

Protect SSH from brute force and services from floods

Project 04 of 06
SSH Brute Force Protection & Rate Limiting
Use the `recent` module to limit SSH to 3 new connections per minute per IP. Add ICMP flood protection with the limit module. Log blocked attempts. This is the standard approach to harden any public-facing SSH server.
Intermediate~20 minfilter table · recent + limit modules

Steps

1

Clean start + safety net

sudo iptables -F && sudo iptables -P INPUT ACCEPT (sleep 90 && sudo iptables -F && sudo iptables -P INPUT ACCEPT) &
2

Allow loopback and established connections

sudo iptables -A INPUT -i lo -j ACCEPT sudo iptables -A INPUT -m state --state ESTABLISHED,RELATED -j ACCEPT
3

Track SSH connection attempts per IP using the recent module

sudo iptables -A INPUT -p tcp --dport 22 -m state --state NEW -m recent --set --name SSH_GUARD
4

Log IPs that exceed 4 connections in 60 seconds

sudo iptables -A INPUT -p tcp --dport 22 -m state --state NEW -m recent --update --seconds 60 --hitcount 4 --name SSH_GUARD -j LOG --log-prefix "SSH-RATELIMIT: "
5

Block IPs that exceed the rate limit

sudo iptables -A INPUT -p tcp --dport 22 -m state --state NEW -m recent --update --seconds 60 --hitcount 4 --name SSH_GUARD -j DROP
6

Allow SSH for IPs under the limit

sudo iptables -A INPUT -p tcp --dport 22 -j ACCEPT
7

Rate limit ICMP to prevent ping floods (5 per second, burst of 10)

sudo iptables -A INPUT -p icmp -m limit --limit 5/sec --limit-burst 10 -j ACCEPT sudo iptables -A INPUT -p icmp -j DROP
8

Set default DROP

sudo iptables -P INPUT DROP

Verify & Test

✅ Test rate limiting (run SSH 5 times fast)
for i in 1 2 3 4 5; do nc -zv -w1 localhost 22 2>&1; done cat /proc/net/xt_recent/SSH_GUARD sudo dmesg | grep SSH-RATELIMIT
✅ How it works
The recent module maintains a per-IP hit list in kernel memory. Rule 3 adds IP to the list on each new SYN. Rules 4+5 check if the same IP has hit more than 4 times in 60 seconds — if so, log and drop. Under the limit → falls through to rule 6 (ACCEPT).

Cleanup

sudo iptables -F && sudo iptables -P INPUT ACCEPT

What you learn

The recent module tracks per-IP hit counts — stateful and very efficient (no conntrack entries for dropped packets)
LOG before DROP lets you see exactly what you're blocking in syslog
The limit module handles burst allowance elegantly — 10 pings burst, then max 5/sec sustained
Rules 3, 4, 5, 6 form a chain: track → if over limit LOG → if over limit DROP → otherwise ACCEPT

Project 5: Dual-ISP Routing

Policy-based routing to fix asymmetric routing with two ISPs

Project 05 of 06
Dual-ISP Policy-Based Routing
Configure two routing tables (one per ISP) and ip rules to ensure traffic arriving via ISP2 replies via ISP2 — not through ISP1 (which would cause asymmetric routing and dropped packets). Use iptables MARK to route specific traffic types via specific ISPs.
Advanced~30 minip rule · ip route · mangle MARK

Steps

1

View current routing and rule state

ip route show && ip rule show
2

Name two routing tables (add to rt_tables if not exists)

echo "100 isp1" | sudo tee -a /etc/iproute2/rt_tables echo "200 isp2" | sudo tee -a /etc/iproute2/rt_tables
3

Populate ISP1 table (replace IPs with your actual values)

sudo ip route add default via 1.2.3.1 dev eth0 table isp1 sudo ip route add 1.2.3.0/30 dev eth0 scope link table isp1
4

Populate ISP2 table

sudo ip route add default via 5.6.7.1 dev eth1 table isp2 sudo ip route add 5.6.7.0/30 dev eth1 scope link table isp2
5

Add policy rules — traffic FROM ISP1 IP uses ISP1 table, FROM ISP2 IP uses ISP2 table

sudo ip rule add from 1.2.3.4 table isp1 priority 100 sudo ip rule add from 5.6.7.8 table isp2 priority 200
6

BONUS: Route HTTP via ISP1, HTTPS via ISP2 using MARK

sudo iptables -t mangle -A OUTPUT -p tcp --dport 80 -j MARK --set-mark 1 sudo iptables -t mangle -A OUTPUT -p tcp --dport 443 -j MARK --set-mark 2 sudo ip rule add fwmark 1 table isp1 priority 50 sudo ip rule add fwmark 2 table isp2 priority 60

Verify

✅ Check routing tables and rules
ip rule show ip route show table isp1 ip route show table isp2 ip route get 8.8.8.8 from 1.2.3.4 ip route get 8.8.8.8 from 5.6.7.8 ip route get 8.8.8.8 mark 1

Cleanup

sudo ip rule del from 1.2.3.4 table isp1; sudo ip rule del from 5.6.7.8 table isp2 sudo iptables -t mangle -F

What you learn

Policy routing uses ip rule to select which routing table to consult — runs before the main table
Asymmetric routing (in via ISP2, out via ISP1) causes packets to be dropped by ISP1 (wrong source IP)
ip rule priority matters — lower number = higher priority, checked first
iptables MARK + ip rule = route specific traffic types via specific paths — most powerful PBR pattern

Project 6: VPN Gateway

Configure a full VPN gateway with routing, NAT, and MSS clamping

Project 06 of 06
WireGuard VPN Gateway
Set up a complete VPN gateway: enable forwarding, allow VPN interface traffic, MASQUERADE VPN clients to the internet, add MSS clamping to prevent VPN fragmentation, and configure split-tunnel routing so specific traffic goes via VPN and the rest uses normal routing.
Advanced~35 minall tables · FORWARD + POSTROUTING + mangle

Steps

1

Install WireGuard

sudo apt install -y wireguard
2

Enable IP forwarding (persistent)

echo "net.ipv4.ip_forward=1" | sudo tee -a /etc/sysctl.conf && sudo sysctl -p
3

Allow traffic from VPN interface (wg0) — accept all traffic arriving via VPN tunnel

sudo iptables -A INPUT -i wg0 -j ACCEPT sudo iptables -A FORWARD -i wg0 -j ACCEPT sudo iptables -A FORWARD -o wg0 -j ACCEPT
4

MASQUERADE VPN clients so they appear as this server's IP to the internet

sudo iptables -t nat -A POSTROUTING -s 10.8.0.0/24 -o eth0 -j MASQUERADE # 10.8.0.0/24 = VPN subnet, eth0 = WAN interface
5

MSS clamping — CRITICAL: prevents VPN TCP connections stalling on large file transfers

sudo iptables -t mangle -A FORWARD -o wg0 -p tcp --tcp-flags SYN,RST SYN -j TCPMSS --clamp-mss-to-pmtu sudo iptables -t mangle -A FORWARD -i wg0 -p tcp --tcp-flags SYN,RST SYN -j TCPMSS --clamp-mss-to-pmtu
6

Allow established return traffic

sudo iptables -A FORWARD -m state --state ESTABLISHED,RELATED -j ACCEPT
7

Split tunnel: specific subnets via VPN, internet via normal route

sudo ip route add 10.0.0.0/8 dev wg0 sudo ip route add 172.16.0.0/12 dev wg0 # Internet traffic still uses default route (eth0)

Verify

✅ Check VPN gateway rules
sudo iptables -L FORWARD -n -v sudo iptables -t nat -L POSTROUTING -n -v sudo iptables -t mangle -L FORWARD -n -v ip route show | grep wg0
✅ Why MSS clamping matters
WireGuard adds ~60 bytes of overhead per packet. Standard MTU is 1500. The effective MTU through WireGuard is ~1420. Without MSS clamping, TCP negotiates max segment size assuming 1500 MTU → large packets get fragmented or silently dropped → downloads stall. MSS clamping fixes this during the TCP SYN handshake.

Cleanup

sudo iptables -F && sudo iptables -t nat -F && sudo iptables -t mangle -F

What you learn

VPN interface (wg0) is just another interface — routing and iptables treat it identically to eth0
MSS clamping in mangle FORWARD prevents the #1 VPN issue: downloads stalling on large files
Split tunnel = specific routes via wg0, default route stays on eth0. Full tunnel = default route via wg0
All 4 tables work together: mangle (MSS clamp), nat (MASQUERADE), filter (FORWARD rules), raw (optional NOTRACK)
